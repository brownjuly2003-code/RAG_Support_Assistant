# План: графовый поиск (GraphRAG-lane) с условной активацией по масштабу корпуса

- Дата: 2026-06-05.
- Статус: PLANNED (не начато; очередь после закрытия parent-expansion цикла).
- Контекст: flat-retrieval (BM25+dense+RRF+reranker+structural chunking+parent-expansion)
  на текущем корпусе (201 док / 5589 чанков / ~4.5MB) даёт 96% FULL @ top-5 —
  графовый слой сейчас НЕ нужен. План фиксирует, КОГДА он становится нужен,
  по какому критерию включается и как валидируется.

## Зачем граф вообще

Графовый поиск (entity/relation-граф поверх корпуса, multi-hop traversal)
закрывает класс запросов, который flat-retrieval закрыть не может в принципе:
- **multi-hop**: «какие процессы зависят от X через Y» — ответ собирается из
  цепочки документов, ни один чанк не содержит связку целиком;
- **агрегации по сущности**: «всё, что касается перевозчика Z» — сущность
  размазана по N докам, top-k чанков не покрывают;
- **глобальные вопросы** («какие основные риски в корпусе») — GraphRAG-style
  community summaries.

На малом корпусе эти классы либо отсутствуют, либо закрываются parent-expansion
и query-expansion дешевле на порядки.

## Критерий активации — обоснование метрики

### Почему НЕ «количество документов»

Док — единица произвольного размера: FAQ на 200 символов и регламент на 50
страниц считаются одинаково. Ни стоимость графа, ни его польза от числа доков
не зависят: corpus из 1000 однострочных snippet'ов графа не требует, corpus из
30 переплетённых регламентов по 100KB — требует. Метрика нестабильна к способу
нарезки контента на файлы (один и тот же текст можно сложить в 10 или 1000 md).

### Предлагаемая метрика: чанки (= нормированные символы) + connectivity-probe

**Первичный порог — количество чанков после ингеста.** Обоснование:

1. Чанк — это и есть «символы, нормированные на chunk_size» (~800):
   N_chunks ≈ total_chars / 800. То есть это символьная метрика, как и
   предлагалось, но в единицах, которыми пайплайн уже оперирует.
2. **Стоимость графа линейна по чанкам**, не по докам: entity/relation
   extraction = 1 LLM-вызов на чанк. 5589 чанков = 5589 вызовов — на free-tier
   это и есть реальный лимитирующий фактор.
3. **Польза графа тоже растёт с чанками**: дилюция flat-retrieval (целевой чанк
   тонет в пуле кандидатов) — функция размера индекса, не числа файлов.
4. Метрика уже вычисляется при каждом ингесте, детерминирована, видна в логах
   и `/metrics`.

**Вторичный гейт — connectivity-probe (польза, а не размер).** Размер сам по
себе не доказывает, что граф поможет: 50k чанков несвязанных FAQ графу нечего
дать. Перед включением — дешёвая проба: entity-extraction на случайной выборке
(~200 чанков, 200 LLM-вызовов), посчитать **долю сущностей, встречающихся в
≥2 документах** (cross-doc edge density). Если связность низкая — граф не
включаем независимо от размера.

**Третий сигнал — eval-based (самый честный):** доля multi-hop/aggregation
кейсов в curated-наборе со статусом MISS/PART при текущем flat-стеке. Граф —
лекарство от конкретной болезни; если болезни в eval нет, порог размера
не основание включать.

### Пороговые значения (стартовые, уточняются по probe)

| сигнал | порог | основание |
|---|---|---|
| chunks после ингеста | ≥ 20 000 (~16MB текста, ~4× текущего) | ниже — parent-expansion/query-expansion дешевле; extraction 20k вызовов ≈ верх дневного free-tier бюджета — раньше этой точки граф не окупает ингест |
| cross-doc entity share (probe) | ≥ 15% сущностей в ≥2 доках | ниже — граф вырождается в per-doc индекс, который уже есть |
| multi-hop MISS в eval | ≥ 5 кейсов класса multi-hop/aggregation MISS | болезнь должна существовать до лекарства |

Активация: `RAG_GRAPH_RETRIEVAL=auto|on|off` (default `off`); `auto` = включён,
когда выполнены порог чанков И connectivity-гейт (пороги — settings:
`RAG_GRAPH_MIN_CHUNKS`, `RAG_GRAPH_MIN_CROSSDOC_SHARE`). Решение логируется при
ингесте с фактическими значениями метрик.

## Фазы (паттерн cheap-first, как в retrieval-barrier плане)

- **Phase 0 — eval-bucket (локально, без моделей).** Разметить curated-кейсы
  тегом `multi_hop`/`aggregation`; снять текущий coverage этого bucket'а на
  flat-стеке. Если bucket пуст — добавить 5-10 синтетических multi-hop кейсов
  по аналогии с aircargo-набором. Выход: baseline болезни.
- **Phase 1 — connectivity-probe (локально, ~200 LLM-вызовов free-tier).**
  Скрипт `scripts/graph_probe.py`: выборка чанков → extraction (mistral-small,
  structured output) → cross-doc share + топ связующих сущностей. Выход:
  go/no-go по пользе и калибровка порогов таблицы выше.
- **Phase 2 — прототип graph-lane (за флагом, default off).** Extraction при
  ингесте → граф в networkx (in-repo, без новой инфраструктуры; персистентность
  pickle/parquet рядом с Chroma) → retrieval-lane: entity-match запроса →
  1-2-hop соседство → чанки этих узлов как третий список в RRF
  (`_rrf_merge_multiple` уже есть). Реранкер общий. Hybrid-путь не ломается —
  graph-lane строго аддитивна, как parent-expansion.
- **Phase 3 — A/B по паттерну Phase 2/D.** kw-coverage (multi-hop bucket
  отдельно) + R7 LLM-judge vs текущий стек; решение о `auto`-дефолте по
  критерию «bucket закрывается, faithfulness не просел».

## Анти-риски

- Не включать графом дефолт без Phase 3 (no shipping blind).
- Extraction на полном корпусе — только после положительного probe (бюджет
  free-tier; на Windows не держать граф+эмбеддер в одном процессе).
- networkx до ~10⁵ узлов достаточен; Neo4j/внешний graph-store — вне скоупа,
  карты/инфраструктуры нет.
- Метрика «чанки» зависит от chunk_size: порог зафиксирован для 800/200;
  при смене нарезки пересчитать как chars/800.
