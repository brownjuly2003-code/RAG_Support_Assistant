# План: доказательство (или замена) chunk_size=800 / chunk_overlap=200

- Дата: 2026-06-05.
- Статус: PLANNED. Очередь — после закрытия parent-expansion цикла (плечо D),
  т.к. сравнивать конфиги нарезки честно только поверх итогового стека.
- Проблема: 800/200 — унаследованные дефолты (типовые для LangChain), ни разу
  не проверенные A/B на этом корпусе. При этом на них завязаны связанные
  параметры (`parent_expansion_max_chars`=3×cap, нормировка порога графового
  плана chars/800).

## Что 800/200 реально контролируют при structural-дефолте

Замер по aircargo-корпусу (201 док, воспроизводимо, без моделей —
`MarkdownHeaderTextSplitter` + распределение длин секций):

| | значение |
|---|---|
| секций всего | 4509 |
| медиана длины | 563 chars |
| p90 / p99 / max | 970 / 1936 / 4044 |

| cap | секций режется | доля |
|---|---|---|
| 400 | 3204 | 71.1% |
| **800 (тек.)** | **827** | **18.3%** |
| 1200 | 183 | 4.1% |
| 1600 | 75 | 1.7% |

Выводы для дизайна эксперимента:
1. С `RAG_STRUCTURAL_CHUNKING=true` (prod default) chunk_size — это **cap на
   секцию**, а не размер чанка: 82% секций он не трогает вовсе. Медианный чанк
   и так 563. Blast radius параметра — только хвост распределения.
2. `chunk_overlap` применяется ещё уже: только внутри char-fallback для
   oversized-секций и для не-markdown доков. Это не «перекрытие всех чанков».
3. cap=400 заведомо плох (режет 71% секций — ровно тот механизм потери
   kw-связок, который Phase 1 доказала для обрезки тел). Из сетки исключён.
4. Главный открытый вопрос — **вверх, не вниз**: cap=1200 оставляет целыми 96%
   секций (vs 82%), т.е. меньше разрезанных field-таблиц — задокументированный
   режим потерь. Цена — более длинные тексты на реранк/в контекст.

## Почему НЕ прокси-A/B (важно)

MiniLM-прокси из Phase 1 здесь неприменим: max_seq=128 токенов ≈ первые
~400-500 символов чанка. Плечи, различающиеся хвостом чанка (800 vs 1200 vs
1600), для прокси-эмбеддера неразличимы — sweep покажет ложный «нет эффекта».
(Смещение задокументировано ещё в отчёте Phase 1.) Поэтому сразу
production-стек по проверенному Kaggle CPU-kernel паттерну.

## Фазы

- **Phase 0 — локальный текст-анализ (сделано выше, дополнить).** Для каждого
  кандидата cap: сколько kw-связок из 100 curated-кейсов перестают резаться
  (та же co-occur методика, что в residual-miss диагнозе). Если cap=1200 не
  возвращает ни одной связки — производственный sweep не нужен, 800 остаётся
  с этим обоснованием. Без моделей, <1GiB.
- **Phase 1 — production sweep (Kaggle CPU kernel, паттерн Phase 2).** Плечи:
  S800=текущий C-конфиг (уже посчитан), S1200, S1600 — поверх полного стека
  (structural + contextual headers + reranker; parent-expansion применяется
  локально post-hoc, как в плече D). Бюджет: ~2.5h encode/плечо на 4 vCPU →
  2 новых плеча + rerank укладываются в 12h-лимит одного kernel.
  Метрики: kw-coverage @ top-5 + матрица переходов (регрессии обязательны).
- **Phase 2 — overlap-чувствительность (только если Phase 1 меняет cap).**
  При фиксированном победителе cap: overlap {100, 200, 400} — эффект только
  через oversized-fallback, ожидаемо малый; если Phase 0 показывает <10
  затронутых кейсов — фазу скипнуть с этим обоснованием.
- **Phase 3 — R7 LLM-judge победителя vs текущего** (mistral free-tier,
  локально, паттерн cont.14) + решение. Критерий: coverage не ниже И
  faithfulness не просел. Любая смена cap тянет пересчёт связанных параметров:
  `parent_expansion_max_chars` (держать 3×cap), нормировка графового порога
  (chars/cap), `RAG_PARENT_EXPANSION_WINDOW` (окно в секциях не меняется).

## Возможный исход «800 доказан»

Отрицательный результат — тоже результат: если Phase 0 показывает, что
oversized-хвост не пересекается с kw-связками кейсов, фиксируем «800/200
обоснован для этого корпуса: cap режет 18% секций, из них 0 несут связки» в
отчёт и закрываем вопрос без траты remote-ресурса. Документ-обоснование —
обязательный выход плана в любом исходе.
